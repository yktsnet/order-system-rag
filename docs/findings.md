# Findings

ベースライン測定の記録。「RAG をどれだけ技術的に使ったと言えるか」を自己評価する過程で、Hybrid 検索や評価指標といった追加機能を足す前に、ベースラインの Retrieval を手動で測定した。想定していた弱点（ベクトル検索が似た文書を混同する）はほぼ起きておらず、代わりに想定していなかった種類の問題が見つかった。「先進的な技術を足して精度を上げる」より「測ったら土台に不具合があった、直したら改善した」という優先順位で進める根拠になった記録である。

各問題は修正済み。README の [Capability Comparison](../README.md#capability-comparison) はこれらの修正後に実施した比較。

## 1. doc_type が実態を表していない（修正済み・Issue 11）

**何を見たか**: 検索結果の `doc_type` が `delivery_09.pdf` のような納品書でも常に `"invoice"` になっていた。

**原因**: `src/ingest/extract.py` の `doc.doc_type` は Azure Document Intelligence の `prebuilt-invoice` モデルが返す固定ラベルであり、見積書/納品書/請求書を実際に分類した値ではない。

**何がわかったか**: 「メタデータフィルタが未接続で弱い」という当初の予想は不十分だった。接続しても当時のままでは機能しない（`doc_type` で絞ると全件ヒットする）ことが、先に直すべき土台の問題として判明した。ファイル名 prefix からの再分類で修正した。

## 2. メタデータフィルタが未接続（修正済み・Issue 12）

**何を見たか**: `src/generate/rag.py` の `_search()` はベクトル検索のみで、インデックスに定義済みの `filter` 可能フィールド（日付・取引先名）を一切使っていなかった。

**実測**: 実在する日付で「2026年6月8日に納品した内容を教えて」と質問すると正解は rank1 に来たが、rank2 とのスコア差はほぼゼロ（0.7927 vs 0.792）。日付という決定的な手がかりがベクトル検索ではほとんど効いていない。

**何がわかったか**: `invoice_date` 等は `doc_type` と違い正しい値が入っているため、フィルタを繋げば確実に・測定可能な形で改善する。Hybrid 検索より費用対効果が高い。`_search()` にフィルタを接続して解消した。

## 3. 曖昧な質問で複数文書が混ざって回答される（対処済み・Issue 16）

**何を見たか**: 「届いた荷物の中身は?」のようにエンティティ（会社名・日付）を含まない質問だと、スコアが僅差の上位5件すべてを回答に混ぜて返してくる。

**何がわかったか**: スコアの近さの問題ではなく、質問に手がかりが無いことが原因。ただし新しい分類レイヤーや聞き返し UI は不要で、既存の無回答ポリシー（`RELEVANCE_THRESHOLD` による refuse）と同じ「どこまで自信を持って答えるか」という生成側のポリシーの話。生成プロンプトに「複数の候補を区別できない場合はその旨を答える」指示を足して対処した。

## 4. ルーティングが分類するだけで分岐していない（修正済み・Issue 14）

**何を見たか**: `route_query` ノードは質問を `sql`/`rag`/`both` に分類していたが、後続のグラフは分類結果に関わらず常に `embed_query → search_docs` の RAG 経路のみを通っていた。

**何がわかったか**: 「2種類を組み合わせている設計は良いモデルでは」という問いに対し、設計思想としては正しいが実装はそれを体現していないという評価になった。分類ロジック自体（LLM 構造化出力）は維持し、`add_conditional_edges` で欠けていた分岐の配線を足した。あわせて分類を2値に整理した（[design-decisions.md](design-decisions.md) 参照）。

## 5. 既存発注 DB とのクロスオリジン問題（方針変更・Issue 13〜15）

**何を見たか**: SearchTab が叩いていた `order-system-migration` の `Orders` テーブルは、`generate_seed.py` で独立生成された合成データ（400件）。会社名（東京商事等）は似せてあるが取引内容は本リポの PDF とは無関係だった。

**何がわかったか**: 旧計画が想定していた「既存の発注 DB と同じ取引先・商品を使い比較可能にする」は実現していなかった。加えて `Orders` はカテゴリ別集計向けのフラットな注文記録で、本リポが扱う「見積→納品→請求」という3段階の文書ライフサイクルとはデータモデルが別物。SQL 側は流用せず、本リポの抽出データに合わせた SQL スキーマを自前で用意する方針に変更した（判断の全文は [design-decisions.md](design-decisions.md)）。
